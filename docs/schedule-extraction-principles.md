# Schedule Extraction Principles

## 目的

スポーツジム各ブランドのレッスン表を、店舗固有ルールに依存せず取得する。

- 店舗トップ URL やブランド配下ページから、スケジュール一覧・詳細・PDF に到達できること
- JEXER だけでなく、他ブランドのジムにも流用しやすい土台にすること
- 個店ごとの場当たり対応ではなく、探索・分類・抽出の原則で改善を進めること

## 非目標

次のような方向には寄せない。

- 店舗ごとのヒント追加
- 店舗専用ハードコード
- 個別店舗の URL 補正ルールを増やし続ける運用

ただし、ブランド全体で共通する構造が明確に観測でき、かつ個店例外ではなく運用上も安定している場合は、ブランド共通 adapter を持つこと自体は許容する。

## 探索の原則

### 1. Same Origin + URLスコア + AIページ分類

- 候補 URL の収集はまず same origin を基本条件にする
- そのうえで URL スコアで有望度を付ける
- AI はページ種別判定と次リンク推薦を担当する

### 2. 一覧ページ中心

- 一覧ページを見つけることを優先する
- `schedule_index` と判定されたページでは、掲載されている詳細候補を広く回収する
- detail を細かく捨てるより、まず取りこぼしを減らす

### 3. recommended_next_links を使った多段探索

- AI は `recommended_next_links` を返す
- 探索は 1 発で正解ページを当てる前提にしない
- 入口 → 一覧 → 詳細 → PDF のような数段の導線を許容する

### 4. 深さ固定ではなく予算制

- `maxVisitedPages`
- `maxAiClassifications`
- `maxPdfPages`
- `maxDepth`

このような予算で探索量を制御する。

目的は、固定的な `depth=1` や `depth=2` に依存せず、コストと網羅性のバランスを取ること。

### 5. ブランド共通 adapter は許容する

- 汎用探索だけで不安定なブランドについては、ブランド共通の discovery strategy や adapter を持ってよい
- ただし許容されるのは、ブランド全体で再利用できる共通導線だけ
- 店舗ごとの URL 補正や、個店ごとの例外分岐は入れない

例:

- 許容: `JEXER は shared schedule endpoint を正規ルートとして扱う`
- 不許容: `四ツ谷だけこの URL を叩く`

### 6. JEXER は shared schedule endpoint を正規ルートとして扱う

JEXER については、汎用探索よりも次のブランド共通構造を優先する。

- 全店舗のスケジュールは `https://www.jexer.jp/schedule/fitness/?shop=<id>` で管理される
- `shop` が店舗 ID である
- したがって JEXER では、店舗ごとに `shopId` を管理し、shared schedule endpoint を直接取得する
- `mb/...` の旧導線探索は補助経路であり、正規ルートではない

この方針は店舗固有ルールではなく、JEXER ブランド全体に共通する adapter として扱う。

## 抽出の原則

### 1. `schedule_detail` / `pdf_schedule` を第一級対象にする

- 抽出対象は detail ページと PDF を同格に扱う
- HTML だけを前提にしない

### 2. HTML と PDF を分けて扱う

- HTML は HTML 用の取得・分類・抽出を行う
- PDF は PDF URL 判定、PDF本文取得、PDF専用抽出を別経路で扱う

### 3. detail 判定は table だけに依存しない

detail 判定は次の複合条件で考える。

- table の有無
- 時刻文字列の有無
- 繰り返し構造
- 時刻 + テキストの密度
- 一覧ページらしいリンク密度との対比

### 4. ページ選定とページ内フィルタを分ける

- `ページ選定` は「どのページを抽出対象にするか」を決める
- `ページ内フィルタ` は「抽出した各枠のうち、どれを通常レッスンとして残すか」を決める
- ページファイル名や URL パターンだけで通常レッスン判定をしない

ページ内フィルタでは、少なくとも次を区別する。

- `regular_class`
- `support_session`
- `personal_session`
- `school_course`
- `member_guidance`

GymMap に入れるのは `regular_class` だけにする。

## AI の役割

AI の役割は次の 3 つに限定する。

### 1. ページ種別判定

- `entry`
- `schedule_index`
- `schedule_detail`
- `pdf_schedule`
- `instructor`
- `ignore`

### 2. 次リンク推薦

- `recommended_next_links` を返し、探索の次段を案内する
- ただし候補収集自体はルールベースでも補完する

### 3. 本抽出

- detail HTML または PDF本文から構造化 JSON を抽出する

### 4. ページ内ブロック判定

- AI は各枠について、通常レッスンか、サポート枠か、パーソナルか、スクールか、会員向け案内かの候補を返してよい
- ただし最終的に残すかどうかは、ルールベースの entry type classifier と組み合わせて決める
- 目的は、`マンツーマンサポート` や `会員向け案内` のようなノイズを通常レッスンから外すこと

### 前提

- AI は店舗固有知識に依存しない
- AI はブランド専用ルールの置き場ではない
- AI は探索ナビゲータ兼抽出器であり、個店専用設定の代替ではない

## 禁止事項

次を積み増さない。

- 店舗別ヒント追加
- 店舗別 URL 補正
- 「この店舗だけこうする」という例外処理の蓄積

ブランド共通 adapter 自体は禁止しない。ただし、次の形は許容しない。

- ブランド名を理由に個店例外を積み増す
- adapter の中で店舗別分岐を増やす
- ブランド共通のはずの adapter を、実質的に個店対応の寄せ集めにする

例外対応が必要に見える場合も、まずは探索原則・スコア・ページ分類・予算設計で吸収できないかを先に検討する。

## Debug / 観測性

探索と抽出は、後から原因を分解できることを前提に設計する。

### `candidate-pages`

用途:

- 入口から何が候補として集まったかを見る
- 候補不足か、候補はあるのに選定に負けたのかを切り分ける

### `selection`

用途:

- 最終的にどのページを抽出対象に選んだかを見る
- `schedule_kind`、URLスコア、構造シグナルの影響を確認する
- ブランド共通 adapter を使った場合は、その adapter で直接選ばれたかどうかも確認する

### `failure`

用途:

- 失敗が fetch・分類・選定・抽出のどこかを分けて記録する
- 候補 fetch 失敗が全体失敗なのか、局所失敗なのかを切り分ける

### `classification`

用途:

- AI が各ページをどう見たかを確認する
- `page_type`、`recommended_next_links`、`confidence` の妥当性を見る

### `gemini input / output`

用途:

- AI に何を渡したか確認する
- HTML / PDF本文が正しく渡っているかを見る
- AI が拾えていないのか、入力が悪いのかを切り分ける

### `extracted-entry-types`

用途:

- 抽出後の各枠が `regular_class / support_session / personal_session / school_course / member_guidance` のどれに寄ったかを見る
- 通常レッスンとして残した件数と、除外したノイズ枠の件数を確認する
- ページ選定ではなく、ページ内フィルタの精度を改善するために使う

## 今回の改善内容の要約

- branch 依存を減らした
- URL スコア制を導入した
- PDF 本文対応を入れた
- prompt を一般化した
- fixed depth から予算制へ寄せた
- detail 判定に繰り返し構造と時刻密度を入れた
- JEXER には shared schedule endpoint を正規ルートとするブランド共通 adapter を入れた
- 抽出後の各枠に entry type 判定を入れ、通常レッスンだけを残すようにした

## 今後の改善優先順位

1. 候補スコア改善
   - URL スコアの加点・減点の精度を上げる
2. PDF 精度改善
   - PDF 本文の整形、抽出精度、表の取り扱いを改善する
3. ページ内フィルタ改善
   - 通常レッスンとサポート枠の判定精度を高める
4. detail 判定改善
   - 構造シグナルの精度を高め、一覧ページとの誤判定を減らす
5. 入口 URL / adapter の持ち方の一般化
   - 将来的にはブランドや店舗 target の持ち方自体もより汎用化する

## 運用メモ

今後 Codex に改善依頼するときは、この文書を基準にする。

- まずこの原則に反していないかを確認する
- 個店対応ではなく、原則側の改善で解けないかを先に考える
- 新しい提案がある場合は、この文書のどの節を更新するかまで含めて扱う
